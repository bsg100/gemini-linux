## Key Findings

1. **Nearly every subsystem in Kali and Gemian uses an Android/MTK vendor driver.** Both projects run the 3.18 MediaTek BSP kernel with minimal Linux-specific changes. The only subsystem with a genuine downstream Linux driver is the **keyboard matrix** (Gemian), which is the best porting reference in the project.

2. **Core boot stack is well covered upstream.** CPU, GIC, timers, clocks, pinctrl, PMIC wrapper, eMMC (MSDC), UART, thermal, and watchdog all have mainline drivers that are clean rewrites of the vendor originals. Do not port from 3.18 — use mainline directly.

3. **RT5735 buck regulators are the main boot-critical gap.** These supply VPROC (CPU voltage) and VGPU. Without a working regulator driver the kernel cannot perform DVFS. For initial bring-up: run at a fixed voltage (skip regulator registration) or port the driver. Check register compatibility with mainlined Richtek parts first.

4. **Wi-Fi/BT is deferred to Phase 9.** MT6797 CONSYS (integrated Wi-Fi/BT/GPS) is entirely proprietary. Architecture confirmed: WiFi uses AHB bus (not SDIO), BT uses BTIF/STP. conninfra, mt76, and btmtksdio are all inapplicable. The vendor driver stack is ~75–103 KLOC and requires a userspace daemon (`wmt_launcher`). The last known working out-of-tree port (frank-w BPI-Router-Linux) ran on kernel 5.6 and is broken on 6.0+. USB-Ethernet (USB dongle via `xhci-mtk`) covers Phase 8 networking. See [research.md](research.md) for full findings.

5. **Display requires significant work and MT6797 has no upstream DRM path.** The mainline MTK DRM driver supports mt2701/mt8173/mt8183/mt8186/mt8195 — MT6797 is absent from every component. MMSYS clocks exist in `clk-mt6797-mm.c`. At minimum, four files require new MT6797 variants: DDP topology (`mtk_drm_drv.c`), MIPITX PHY (`phy-mtk-mipi-dsi.c`), DSI controller (`mtk_dsi.c`), and display pipeline DTS nodes in `mt6797.dtsi`. The R63419 panel init sequence is fully documented in vendor DTS/LCM source. Display bias IC identity is unknown and needs hardware investigation before display bring-up can proceed.

6. **AW9523B GPIO driver is NOT in mainline** — the prior note claiming "mainlined ~6.0" is incorrect (no `aw9523` entries found in `drivers/gpio/` in Linux 6.6). This changes the keyboard plan: `matrix_keypad.c` exists and uses standard gpiolib, but requires an AW9523B GPIO driver first. Two paths: write AW9523B GPIO driver + use `matrix_keypad` (clean, upstreamable), or port Gemian's custom input driver directly (most direct). Gemian's driver remains the best reference implementation.

7. **Sensor driver crash risk.** STK3x1x, LSM6DS3, and BMI160 vendor drivers caused crash loops on prior 3.18 Linux boot (I2C probe before hardware ready). On mainline with DT-driven probing this should not recur, but these drivers should be modules or excluded until the system is stable. Use IIO drivers, not vendor sensor hub.

8. ~~RT9466 charger uses `richtek,rt9467` binding.~~ **CORRECTION (2026-07-14): the charger is a TI BQ25896 at i2c0 0x6b, not an RT9466** (verified live via chip-ID register; nothing at 0x53 on any bus — the RT9466 identification came from a dead vendor config branch). Use mainline `bq25890_charger.c` with `compatible = "ti,bq25896"`. Still the simplest use-upstream path; also provides the `usb-otg-vbus` boost regulator for USB host mode. See research.md §8.

9. **USB2 host mode works without FUSB301A.** Basic USB2 (D+/D-, VBUS) does not require the CC controller. Fixed polarity and 500 mA default, but USB networking adapters and host-mode peripherals function. FUSB301A can be deferred to Phase 4+ for a minimal CC-detection driver; full Power Delivery support is not needed.

10. **MT6351 fuel gauge has no mainline support.** No MT6351 ADC or coulomb counter driver exists in Linux 6.6. Use charger-only mode (bq25890_charger.c driver — see charger correction in note 8) plus userspace voltage monitoring for minimum safe operation. Full fuel gauge requires a new `mt6351-adc.c` driver; defer to Phase 7.

11. **CONSYS Wi-Fi porting starting point is kernel 5.6, not 3.18.** The frank-w BPI-Router-Linux project already ported `wlan_drv_gen2` to kernel 5.6 (last known working). The breakage at 5.7 (BT) and 6.0 (WiFi) is the actual gap to close — not a full 3.18→6.6 leap. Starting from the frank-w 5.6 tree and identifying the 5.7/6.0 API breakages is the correct approach if this is ever attempted. Note: BPI-R2 uses external SDIO MT6625 (different HIF than MT6797 AHB), so the wifi gen2 driver core is shared but the HIF layer would need MT6797 AHB path work. USB-Ethernet remains the Phase 8 plan.

12. **Recommended minimum kernel version: 6.1 LTS** — mature Panfrost T860, MT6797 DTS support, and the bq25890 charger driver (BQ25896 support; charger identity corrected 2026-07-14, note 8). AW9523B GPIO driver is not in mainline; keyboard port is required. ~~covers MT6351 regulator (6.2)~~ **CORRECTION (2026-06-10): the MT6351 regulator never landed in mainline at any version** — verified against the v6.6 tree (only the ASoC codec exists). See PMIC row.

13. **MT6351 PMIC is NOT upstreamed (correction, 2026-06-10).** Earlier rows claimed the regulator and RTC were mainline; direct inspection of v6.6 shows no MT6351 MFD/regulator/RTC anywhere — only `sound/soc/codecs/mt6351.c`. Bring-up consequence: all MT6351 rails must be `regulator-fixed` stubs (LK leaves them configured); eMMC `vmmc`/`vqmmc` stubs needed for Phase 4; RTC deferred. A real MT6351 MFD+regulator port (mt6397-family pattern) is a new post-freeze driver-port item.

## Device Tree Starting Point

The vendor DTS (`compatible = "mediatek,MT6797"`) provides the hardware register map. The mainline DTS file `arch/arm64/boot/dts/mediatek/mt6797.dtsi` provides the upstream-compatible node structure. The bring-up DTS should be built from the mainline dtsi, adding Gemini-specific nodes (board file) for the peripherals confirmed above.

Vendor DTB decompiled to [`docs/vendor-dtb/gemini_kali_boot.dts`](docs/vendor-dtb/gemini_kali_boot.dts) (6098 lines, committed 2026-06-10; extracted from `planet/kali_boot.img`) — the reference for DTS authoring. The raw DTB and the known-good kernel's extracted config are alongside it.
